## 사용법

사용자가 커밋을 요청하면:

1. `git status`와 `git diff`로 변경사항을 분석한다
2. 변경 성격에 맞는 type과 메시지를 결정한다
3. 사용자에게 커밋 메시지를 확인받는다
4. `git commit`으로 커밋한다

## 커밋 메시지 규칙

```
{type}: {변경 내용 요약}
```

- 제목은 한국어, 명령형/요약형으로 간결하게
- 한 커밋은 하나의 논리적 변경만 담는다
- 변경 범위가 크면 type별로 나눠 커밋을 제안한다

## 타입 매핑

| 변경 내용     | 타입       | 예시                                  |
| ------------- | ---------- | ------------------------------------- |
| 새 기능       | `feat`     | feat: 소셜 로그인 기능 추가           |
| 버그 수정     | `fix`      | fix: 카드 컴포넌트 렌더링 오류 수정   |
| 스타일 변경   | `style`    | style: 버튼 여백 조정                 |
| 리팩토링      | `refactor` | refactor: 모임 목록 컴포넌트 분리     |
| 설정/의존성   | `chore`    | chore: ESLint 설정 추가               |
| 문서          | `docs`     | docs: README 업데이트                 |

## 실행

```bash
# 1. 변경사항 확인
git status
git diff --staged

# 2. 스테이징 (필요 시)
git add {files}

# 3. 커밋
git commit -m "{type}: {내용}"
```

## 주의사항

- 커밋 전 변경 파일과 메시지를 사용자에게 확인받는다
- AI가 생성한 코드도 컨벤션(`docs/PROJECT_GUIDE.md`) 준수 여부를 확인한다
- 스테이징되지 않은 변경이 있으면 안내한다
- 민감 정보(.env, 키)가 포함되면 경고하고 중단한다
